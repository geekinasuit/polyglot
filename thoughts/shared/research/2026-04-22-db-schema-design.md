# DB Schema Design: bracket_pair

This document is the authoritative DBA output for the DB feature. The Architect agent reads
this first and builds the Kotlin API design on top of it. Each decision is marked **fixed**
(do not deviate without DBA sign-off) or **provisional** (open to refinement with good reason).

## 1. Schema: bracket_pair DDL

**Decision: fixed**

The migration file at `db/migrations/20260421000000_create_bracket_pair.sql` defines:

```sql
-- migrate:up
CREATE TABLE bracket_pair (
    id         INTEGER  PRIMARY KEY,
    open_char  CHAR(1)  NOT NULL,
    close_char CHAR(1)  NOT NULL,
    enabled    BOOLEAN  NOT NULL DEFAULT TRUE
);

INSERT INTO bracket_pair (id, open_char, close_char) VALUES (1, '(', ')');
INSERT INTO bracket_pair (id, open_char, close_char) VALUES (2, '[', ']');
INSERT INTO bracket_pair (id, open_char, close_char) VALUES (3, '{', '}');

-- migrate:down
```

This DDL is confirmed correct. No corrections needed.

### Column-by-column portability analysis

**`id INTEGER PRIMARY KEY`**

- SQLite: `INTEGER PRIMARY KEY` is the SQLite auto-increment rowid alias. When explicit values
  are provided in INSERT (as the seed data does), SQLite stores them as-is. No auto-increment
  conflict arises because we always supply explicit IDs.
- PostgreSQL: `INTEGER PRIMARY KEY` is a plain integer with a unique constraint. Explicit IDs
  work identically.
- H2: `INTEGER PRIMARY KEY` is supported natively. H2 accepts explicit IDs with no additional
  configuration.
- Conclusion: safe across all three. The decision to use explicit `id` values in seed inserts
  (rather than relying on auto-increment) avoids the `SERIAL` vs. `GENERATED ALWAYS` conflict.

**`CHAR(1) NOT NULL`**

- SQLite: SQLite has text affinity for `CHAR(n)` — the `(1)` is advisory, not enforced at the
  storage level. `CHAR(1)` parses and stores correctly; single-character enforcement is an
  application-layer concern.
- PostgreSQL: `CHAR(1)` is a fixed-width character type; values longer than 1 character are
  rejected. This provides a weak structural constraint at the DB layer.
- H2: `CHAR(1)` is supported natively with the same semantics as PostgreSQL in H2's default
  mode.
- Conclusion: safe across all three. The application layer (JOOQ + repository) enforces
  single-character values; relying on DB enforcement would be PostgreSQL-only anyway.

**`BOOLEAN NOT NULL DEFAULT TRUE`**

- SQLite: SQLite has no native BOOLEAN type; it stores booleans as INTEGER (0/1). SQLite
  accepts `BOOLEAN` as a type affinity name and maps it to INTEGER. `DEFAULT TRUE` is parsed
  and stored as 1.
- PostgreSQL: native BOOLEAN type; `NOT NULL DEFAULT TRUE` works as expected.
- H2: native BOOLEAN type; `NOT NULL DEFAULT TRUE` works as expected.
- Conclusion: safe across all three. JOOQ's SQLite dialect correctly maps `BOOLEAN` columns
  to `Boolean` in the generated record. The seed INSERTs omit `enabled`, so all three pairs
  receive the default value `TRUE` (or 1 on SQLite).

### What to avoid (constraint reference for future migrations)

Any future migration must stay within the safe cross-platform subset:

- No `RETURNING` clause (PostgreSQL/H2 only; SQLite does not support it)
- No `SERIAL` or `GENERATED ALWAYS AS` (use explicit IDs or sequences only if pg-only is acceptable)
- No `ADD COLUMN ... AFTER` (SQLite ignores column order; H2 does not support `AFTER`)
- No stored procedures or triggers
- No `ON CONFLICT DO UPDATE` (PostgreSQL upsert syntax; not supported in H2 default mode)

## 2. Migration Filename Convention

**Decision: fixed**

The timestamp-format filename `20260421000000_create_bracket_pair.sql` is valid for both
Flyway and dbmate when Flyway is configured with:

```kotlin
Flyway.configure()
    .sqlMigrationPrefix("")       // no "V" prefix
    .sqlMigrationSeparator("_")   // single underscore separates version from description
```

**Evidence:** This was validated in PR 2 (DB-001 implementation). The `db/README.md`
documents the validated configuration, and the `db/schema.sql` file (generated by dbmate
as the canonical schema snapshot) shows the migration applied successfully:

```sql
-- Dbmate schema migrations
INSERT INTO "schema_migrations" (version) VALUES
  ('20260421000000');
```

Flyway parses `20260421000000_create_bracket_pair.sql` as:
- Version: `20260421000000`
- Description: `create bracket pair`

dbmate accepts this format natively (it is dbmate's own convention).

The fallback (separate `flyway/` and `dbmate/` subdirectories) is not needed and was not
implemented. All migration files share a single `migrations/` directory.

## 3. Seed Data Strategy

**Decision: fixed**

The three standard bracket pairs `( )`, `[ ]`, `{ }` are inserted directly in the migration
file (`20260421000000_create_bracket_pair.sql`), not in a separate seed mechanism.

**Rationale:**

These are reference data, not user data. Every environment — local dev, CI, production —
requires them to exist on first startup. Embedding them in the migration guarantees that
Flyway (and dbmate) applies them atomically with the schema creation. There is no environment
where the table should exist but the three standard pairs should not.

**Test implications:** H2 in-memory databases are created fresh for each test class (see
section 6). A test that wants to start with only the seeded pairs can simply run Flyway on
a new in-memory DB and use the result as-is. A test that wants different data inserts
additional rows or issues targeted DELETEs in test setup — this is clean and idiomatic.
There is no shared mutable state between test classes.

**Trade-off acknowledged:** If a future requirement allows removing the standard pairs (e.g.,
a deployment that uses only `< >` brackets), those rows would need to be deleted via a new
forward migration — they cannot be excluded at install time. This is acceptable: such a
requirement is speculative, and a migration to remove specific IDs is trivial. The
reproducibility benefit of inline seed data outweighs this.

## 4. Future Schema Evolution

**Decision: provisional** (this section describes extensions, not current state)

The following additions are plausible and have been evaluated for compatibility with the
current schema design. None of them require altering existing columns.

**`sort_order INTEGER`** — controls display/processing order when the pair list is presented
to a user or configuration UI. Add as a nullable or defaulted column in a new migration.
The current design does not foreclose this; existing rows would get a default value or NULL.

**`category VARCHAR`** — groups pairs (e.g., "programming", "math", "prose"). Add as a
nullable `VARCHAR` column. The current schema has no category dimension; adding it is
purely additive.

**`locale VARCHAR`** or **`locale_id` (FK to a locale table)** — associates pairs with a
language or locale context. Add as a nullable `VARCHAR` (BCP 47 tag, e.g., `en`, `zh-Hant`)
or a foreign key to a separate locale table. The current schema is locale-agnostic; this is
additive.

**Per-user configuration** — a `user_bracket_pair` join table (or a `user_id` column with
overrides) that allows per-user enablement. This does not require altering `bracket_pair`;
it requires a new table. The current schema's `id` column serves as a natural FK target.

**What the current design forecloses:** nothing material. The schema is intentionally minimal
(four columns), which maximizes forward flexibility. The only constraint is that `id` is an
`INTEGER PRIMARY KEY` rather than a UUID — this is fine for a small reference table with a
known, bounded set of rows.

## 5. dbmate Compatibility

**Decision: fixed**

The `-- migrate:down` marker at the end of the migration file is required by dbmate even when
the down migration is empty. dbmate requires both `-- migrate:up` and `-- migrate:down`
markers to be present in the file.

- **dbmate** interprets `-- migrate:up` as the start of the forward migration block and
  `-- migrate:down` as the start of the rollback block. An empty `-- migrate:down` is valid
  and means "no rollback defined."
- **Flyway** interprets both markers as plain SQL comments (they are valid SQL comment syntax:
  `-- text`). Flyway ignores them and runs all non-comment SQL in the file.

The file as written (`-- migrate:up` block with SQL, then `-- migrate:down` at the end with
no SQL below it) is correct for both tools simultaneously.

No separate file format or conversion step is needed.

## 6. Test Data Strategy

**Decision: fixed**

Integration tests use H2 in-memory databases, not SQLite. The pattern:

1. Each test class creates a fresh H2 in-memory DataSource with a unique database name:
   ```
   jdbc:h2:mem:<unique-name>;DB_CLOSE_DELAY=-1
   ```
   `MODE=PostgreSQL` is deliberately omitted. The DDL is written for the cross-platform safe
   subset; PostgreSQL compatibility mode changes H2's type handling and SQL parsing in ways
   that could mask portability bugs rather than surface them. The Architect or TDD Designer
   may add a mode if there is a specific reason (e.g., surfacing PG-specific behavior earlier
   in tests), but it is not required by this schema. This is a **provisional** detail.
2. Flyway is applied to that DataSource in test setup (`@Before` / `beforeEach`), using the
   same migration files from `//db:migrations`. This ensures the exact same schema (including
   seed data) as production.
3. Test-specific rows are inserted in test setup after Flyway runs.
4. No shared mutable state between test classes — each test class owns its in-memory DB.

**Why H2, not SQLite:**
- H2 is already a required dependency for JOOQ's `DDLDatabase` codegen (build-time). Adding
  it to test scope adds no new dependency.
- H2 in-memory databases are purely in-process — no file-system state, no cleanup needed,
  no cross-test contamination.
- SQLite-jdbc would be an additional test-scope dependency that brings no benefit over H2 in
  this context.

**Critical implementation detail — `DB_CLOSE_DELAY=-1`:**

H2 drops an in-memory database when the last connection to it closes. If Flyway and the test
code obtain connections from a HikariCP pool (which opens and closes connections
independently), the database can be dropped between the Flyway run and the first test query.
Setting `DB_CLOSE_DELAY=-1` keeps the in-memory database alive for the lifetime of the JVM,
preventing this. This parameter is **required** in all test JDBC URLs for H2 in-memory
databases used with a connection pool.

**Flyway location for tests:**

Flyway must use `filesystem:` location pointing to the runfiles-resolved path of
`//db:migrations`, not classpath scanning. The same `RUNFILES_DIR` resolution pattern used
in production applies to tests. The filename configuration (`sqlMigrationPrefix` and
`sqlMigrationSeparator`) must also match, or Flyway will default to looking for `V*__*.sql`
and find nothing:

```kotlin
val migrationsPath = Paths.get(
    System.getenv("RUNFILES_DIR") ?: ".",
    "_main/db/migrations"
).toAbsolutePath().toString()

Flyway.configure()
    .sqlMigrationPrefix("")
    .sqlMigrationSeparator("_")
    .locations("filesystem:$migrationsPath")
    .dataSource(testDataSource)
    .load()
    .migrate()
```

Test Bazel targets must declare `data = ["//db:migrations"]` so the runfiles are available.

## 7. Bazel Filegroup

**Decision: fixed**

The existing `db/BUILD.bazel` filegroup is correct and sufficient:

```python
filegroup(
    name = "migrations",
    srcs = glob(["migrations/*.sql"]),
    visibility = ["//visibility:public"],
)
```

**Ordering:** Bazel's `glob()` returns files in lexicographic order. Timestamp-format filenames
(`YYYYMMDDHHMMSS_description.sql`) sort lexicographically in chronological order — this is
the intended property of the timestamp convention. No explicit ordering annotation is needed.

**Subdirectory layout:** There is no subdirectory structure under `migrations/`. All `.sql`
files live directly in `migrations/`. `glob(["migrations/*.sql"])` matches only files in that
directory (not recursive), which is correct. If subdirectories are ever introduced (e.g., for
split migration sets), the glob pattern would need updating to `glob(["migrations/**/*.sql"])`.
For now, the flat layout is correct.

**Edge cases:**
- If a non-migration `.sql` file appears in `migrations/` (e.g., a fixture or snapshot file),
  it would be included in the filegroup and picked up by Flyway. Convention: only migration
  files belong in `migrations/`. Other SQL files (e.g., `db/schema.sql` as generated by
  dbmate) live directly in `db/`, not in `migrations/`.
- The `//db:migrations` filegroup must be added to the `data` attribute (not `deps`) of any
  Bazel target that needs the SQL files at runtime: the service binary, integration test
  targets, and the JOOQ codegen genrule.
